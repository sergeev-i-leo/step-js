Стек SPA

React, Angular, Vue.js, Svelte...
Преимущества:
  • зрелость инструментов
  • большое коммьюнити
  • доступность специалистов на рынке труда
Недостатки:
  • большой 'технический оверхед' и, как следствие...
  • ...крутая кривая обучения
  • постоянные изменения компонент и подходов
  • стоимость поддержки и работы с легаси
4

Стек Клиент-сервер

Express, Flask, Django...
Преимущества:
  • зрелость иструментария
  • большое коммьюнити
  • доступность специалистов на рынке труда
Недостатки:
  • непопулярность в enterprise и, как следствие,...
  • ...невысокий уровень специалистов
  • инфраструктурные сложности (нужно обслуживать дополнительный сервер на каждое решение)
  • простота инструментария проста только на первый взляд: большое количество артефактов (настроек, теплейтов, включений и т.п.)
5

Стек Java

Vaadin, Jimix...
Преимущества:
  • фулл-стек разработка
  • низкая стоимость
  • отсутствие архитектуры
Недостатки:
  • технология 'перпендикулярна' стандартам отрасли
  • необходимость изучать диалект html и css, и, как следствие...
  • ...низкая мотивация разработчиков
  • ограничения технологии и проблемы с кастомизацией
6

Step JS

Отношение цена / качество
Преимущества:
  • соответствует стандартам отрасли (SPA-приложение Typescript)
  • упрощённая архитектура React
  • реализация архитектуры Middleware + Flux
  • поддержка 'из коробки' построения микро-фрондэнд приложений с физическим разделением кода
  • поддержка 'из коробки' компонент Bootstrap
  • низкий порог вхождения, возможнсть использованиия full-stack разработки
Недостатки:
  • отсутствие коммьюнити
  • требуются базовые знания Typescript
  • требуются базовые знания работы браузера и JavaScript
  • требуются базовые знания вёрстки
7

Экономика использования StepJS

Максимум функциональности за адекватный бюджет
По сравнению с SPA:
  • более, чем двукратное уменьшение программного кода при реализации одинаковой функциональности
  • возможность вёрстки экранов по принципам Java (как в Swing, JavaFX и т.п.)
  • реализация архитектуры Middleware + Flux
  • возможность разделения приложения на назависимые модули и их разработка с полным разделением кода
  • снижение стоимости поддержки за счёт уменьшения программных артефактов (чем меньше кода, тем проще с ним работать)
  • отделение вёрстки от бизнес-логики, с возможностью независимой разработки одного и другого
  • низкий порог вхождения
